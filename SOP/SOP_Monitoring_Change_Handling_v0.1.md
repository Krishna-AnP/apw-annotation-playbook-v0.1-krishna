# SOP – Change Handling (v0.1)

# Purpose
Manage changes to scope, labels, or process without breaking quality.

# Scope
In: Change requests

# Owner Role
Project Operator

## Inputs Needed
- Change request
- Impact analysis

# Outputs Produced
- Approved or rejected change
- Updated documentation

# Tools Touched
- GitHub
- CVAT Cloud

# Timebox Range
15–30 minutes per change

# Failure Modes
- Changes applied informally
- No documentation
- Annotators not informed

# Escalation Rules
Stop work if major change is requested mid-batch

# Assumption Ledger
- Changes are infrequent

# Step-by-Step Procedure
1. Receive change request  
   Expected result: Change documented

2. Analyze impact  
   Expected result: Risks understood

3. Approve or reject change  
   Expected result: Decision recorded

4. Update documents if approved  
   Expected result: Single source of truth

5. Inform team  
   Expected result: No confusion

# Embedded Quality Checks
- Change logged
- Docs updated
- Team acknowledged

# Artifacts Produced
- TPL_Change_Request_v0.1.md

# Example Run
Label change requested and approved

# Failure Case
Unapproved change → rework needed
